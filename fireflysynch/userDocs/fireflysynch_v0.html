Introduction

Realizzazione in qak di una simulazione delle lucciole

Requirements

Costruire il simulatore di un sistema costituito da 3 lucciole. Inizialemente, le lucciole lampeggiamo ciascuna con una
frequenza (costante) propria.

  • A) Dopo 10 secondi le lucciole devono lampeggiare in modo sincronizzato
    • 

  • B) Il sistema è dotato anche di un sonar . Quando il sonar rileva un ostacolo a una distanza minore di una distanza
prefissata DMIN, le lucciole devoono lampeggiare in modo sincronizzato. Quando la distanza dell’ostacolo
(ri)supera DMIN, le lucciole tornano a lampeggiare in modo random
    •

    Requirement analysis

    A seguito dell'analisi sono state individuate due entità fondamentali:

      

    • Le lucciole, che lampeggianoin maniera indipendente l'una dalle altre
      • 

    • Un'entità esterna che funziona da orchestratore e sarà incaricato di gestire la transizione delle lucciole nelle diverse modalità di funzionamento
      • 

    
Inoltre per testare il punto B è stato aggiunto temporaneamente un terzo attore che simula il sonar e invia periodicamente un evento di cambio modalità all'orchestratore.

QActor firefly context ctxfirefly {
	[#var Period = 0L#]
	[#var Phase = 0L#]
    State s0 initial {
    	// Nello stato iniziale si inizializza in maniera randomica
		// la frequenza e la fase della lucciola.
    }
    Transition t0
        whenTimeVar Phase -> flash

     State flash {
        // Nello stato di flash, la lucciola emette un flash
        // e scatena un evento per notificare le altre.
	    println("$name | Flash") color yellow
	}
	Transition t0
	    whenTimeVar Phase -> flash
	    whenEvent changeMode -> onChangeMode
	
	State onChangeMode {
        // Nello stato di onChangeMode, la lucciola cambia modalità di funzionamento
        // a seconda dalla modalità richiesta dall'orchestratore.
	    onMsg(changeMode : changeMode(Mode, Period)){
	    }
	}
	Transition t0
	    whenTimeVar Phase -> flash
	    whenEvent changeMode -> onChangeMode
}


QActor fireflySystem context ctxfirefly {
    State s0 initial {
    }
    Transition t0
        whenEvent obstacleClose -> onObstacleClose
		whenEvent obstacleFar -> onObstacleFar

	State onObstacleClose {
		// Nello stato di onObstacleClose, l'orchestratore riceve un evento di ostacolo vicino
		// e invia un messaggio a tutte le lucciole per farle passare alla modalità sincronizzata.
	    onMsg(obstacleClose : obstacleClose()){
	    	emit changeMode : changeMode("synchronized", 1500)
	    }
	}
	Transition t0
	    whenEvent obstacleClose -> onObstacleClose
		whenEvent obstacleFar -> onObstacleFar

	State onObstacleFar {
		// Nello stato di onObstacleFar, l'orchestratore riceve un evento di ostacolo lontano
		// e invia un messaggio a tutte le lucciole per farle passare alla modalità random.
	    onMsg(obstacleFar : obstacleFar()){
	    	emit changeMode : changeMode("random", 0)
		}
	}
	Transition t0
	    whenEvent obstacleClose -> onObstacleClose
		whenEvent obstacleFar -> onObstacleFar
}

    Problem analysis

    L'interazione tra gli attori è completamente gestito ad eventi in quanto la comunicazione non ha nè bisogno di risposta nè tantomeno di sapere chi è il mittente o destinatario del messaggio.
I messaggi del sonar passano attraverso l'orchestratore che si occupa di inoltrarli alle lucciole, in questo modo l'orchestratore si comporta come livello di astrazione intermedio garantendo
la separazione delle reponsabilità e permette in futuro di modificare sia il sitema sonar che le lucciole in maniera trasparente per l'altro lato.
Una possibile alternativa è quella del sonar di inviare continuamente (in push o pull) la distanza rilevata all'orchestratore, che poi decide se inviare o meno un messaggio di cambio modalità alle lucciole, 
in questo modo si avrebbe una maggiore flessibilità nella scelta dello switch.

Event changeMode : changeMode(Mode, Period)
Event obstacleClose : obstacleClose(Distance)
Event obstacleFar : obstacleFar(Distance)

Context ctxfirefly ip [host="localhost" port=8010]

    Test plans

    Come detto in precedenza il sonar è realizzato da un attore mock per facilitare il test iniziale del sistema:

QActor sonarmock context ctxfirefly {
	[#var RandomTimer = 10000L#]
	State s0 initial {
		[#RandomTimer = java.lang.Math.abs(java.util.Random().nextLong() % 10000L)#]
	}
	Transition t0
		whenTimeVar RandomTimer -> changeMode
	
	State changeMode {
		if [#java.util.Random().nextBoolean()#] {
			emit obstacleClose  : obstacleClose(0)
		} else {
			emit obstacleFar : obstacleFar(100)
		}
	}
	Goto s0
}
